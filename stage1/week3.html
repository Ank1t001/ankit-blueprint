Why Linux? The Operating System of the Cloud
⏱ 1.5 hrs📖 Lesson + Setup🔬 Terminal First Steps
Lesson

Why Linux Matters More Than You Think

You've worked on Windows your whole career. That's fine — but the cloud runs on Linux. Over 95% of AWS EC2 instances, Lambda environments, container runtimes, and managed services run Linux under the hood. When you SSH into a production server to investigate a security incident, it will be Linux. When you write automation scripts, they run on Linux. When security tools like CloudWatch Agent, ClamAV, or Fail2ban collect evidence, they run on Linux.

The good news: you don't need to be a Linux sysadmin to start. You need to be comfortable navigating the file system, managing users and permissions, running processes, and writing basic scripts. That's exactly what this week delivers.

What Linux Is (and Isn't)

Linux is an open-source operating system kernel created by Linus Torvalds in 1991. A "distribution" (distro) is Linux + additional software bundled together. AWS uses Amazon Linux 2, Ubuntu, Red Hat, and others.

The key difference from Windows: Linux is designed to be controlled from the command line. GUIs exist but are rarely used on servers. Everything — file management, user creation, network configuration, service management — is done through commands in a terminal.

Amazon Linux 2023 (AWS default EC2) · Ubuntu 22.04 LTS (developer favourite) · Red Hat Enterprise Linux (enterprise) · Debian (lightweight, stable)
The Linux Philosophy — Everything is a File

In Linux, almost everything is represented as a file — configuration files, devices (hard drives, keyboards), network connections, and system processes all live in the file system. This means the same commands for reading and writing files work for interacting with hardware and processes. It's a unified, powerful model once you understand it.

/etc/passwd — list of user accounts (a text file)
/dev/sda — your hard drive (a special file)
/proc/1234 — running process with PID 1234 (a virtual directory)
🏢 Marketing Analogy

If Windows is like using Meta Ads Manager with its GUI dashboards and buttons, Linux is like using the Meta Ads API directly — more setup to learn, but dramatically more powerful, automatable, and precise once you understand it. Cloud security work requires direct API access. The GUI is for demos; the command line is for real work.

Getting Access to a Linux Terminal — 3 Options

  • 1
    WSL2 on Windows (Recommended): Windows Subsystem for Linux 2. Open PowerShell as Administrator and run wsl --install. Restart. Open "Ubuntu" from Start menu. Full Linux terminal inside Windows — zero performance penalty.
  • 2
    Mac Terminal (Already there): Open Terminal app (Cmd+Space → "Terminal"). Mac runs on Unix, the same family as Linux. Every command in this week works natively on Mac.
  • 3
    Browser-based (No install): Go to killercoda.com → Linux Playground. Full Linux terminal in your browser. Zero installation. Use this if WSL2 isn't an option.
💡 The Terminal Will Feel Uncomfortable at First — That's Normal

Every security engineer felt lost in the terminal the first week. By the end of this week you'll have run 60+ Linux commands. By Week 10, the terminal will feel faster and more natural than any GUI. The only way to learn it is to use it — don't skip the labs.

Hands-On Lab
Lab 1
Open Your Terminal & Run Your First 5 Incident Response Commands
  • 1
    Open your terminal. You'll see a prompt like ankit@hostname:~$. The ~ means home directory. The $ means regular user (not root/admin).
  • 2
    Type whoami and press Enter. Prints your username. During security investigations, the first question is always "what user context am I running as?" — if it says "root" when it shouldn't, that's a red flag.
  • 3
    Type uname -a. Prints the Linux kernel version, OS info, and hostname. On an unfamiliar server, this tells you exactly what you're dealing with before you do anything else.
  • 4
    Type date. Prints current date and time. On a server, timestamps are everything — incident timelines, log correlation, and forensic analysis all depend on accurate server time.
  • 5
    Type uptime. Shows how long the system has been running. A server that rebooted unexpectedly at 3am is a security concern worth investigating.
  • 6
    Type history. Shows all previously run commands in your session history. On a compromised system, this reveals exactly what an attacker did after they got in. Note: sophisticated attackers run history -c to clear it — which is itself a suspicious sign in logs.
  • 7
    Write in your notes: These 5 commands (whoami, uname, date, uptime, history) are the first things you run when SSHing into an unfamiliar server during incident response. Explain in one sentence why each one matters.

Lesson

The Linux Directory Structure — Know Where Things Live

Linux has a single file system tree starting at / (called "root"). Unlike Windows with C:\ D:\, everything hangs off one root. Knowing where things live is essential — configuration files, logs, user data, and system binaries all have specific homes, and during an incident investigation, you'll go straight to the right directory.

DirectoryContainsSecurity Relevance
/etcSystem configuration filespasswd, shadow, sshd_config, sudoers — critical security configs
/var/logSystem and application logsauth.log, syslog, nginx/apache logs — your evidence trail
/homeUser home directoriesUser data, .bash_history, .ssh keys per user
/rootRoot user's homeRoot's .bash_history — attacker's actions if they got root
/tmpTemporary files (world-writable)Attackers frequently drop malware here — monitor closely
/bin /usr/binUser commands/binariesls, cat, grep — standard tools live here
/etc/cron.dScheduled tasksAttackers use cron for persistence — review after incidents

Essential Navigation Commands — Muscle Memory Required

# Where am I? pwd # print working directory → /home/ankit # List files ls # basic list ls -la # long format + hidden files (start with .) ls -lh # human-readable file sizes (KB, MB) ls -lt # sorted by modification time (newest first) # Move around cd /etc # go to /etc cd .. # go up one directory cd ~ # go home cd - # go back to previous directory # Read files cat /etc/passwd # print entire file less /var/log/syslog # scroll (q to quit, /search to find) head -20 file.txt # first 20 lines tail -20 file.txt # last 20 lines tail -f /var/log/auth.log # follow live — watch as new lines appear # Search — forensics fundamentals grep "Failed" /var/log/auth.log # find lines with "Failed" grep -i "error" app.log # case-insensitive grep -r "password" /etc # search recursively in directory find /tmp -type f -mtime -1 # files modified in last 24h find /etc -name "*.conf" 2>/dev/null # all .conf files, hide errors
💡 Real Forensics Commands

During incident response you'll combine these: tail -f /var/log/auth.log to watch logins live, grep "Failed password" /var/log/auth.log | tail -50 to see recent brute force attempts, and find /tmp -type f -newer /etc/passwd to find files dropped after the system was set up. These aren't hypothetical — they're actual commands from real incident responses.

Hands-On Lab
Lab 2
Navigate the Linux File System & Read System Files
  • 1
    Open your terminal. Run pwd — confirm home directory. Run ls -la — note files starting with . (dot files are hidden configuration files).
  • 2
    Navigate to /etc: run cd /etc then ls -la. This directory contains almost all system configuration. Every Linux server has this structure.
  • 3
    Run cat /etc/passwd. This lists all user accounts. Each line format: username:password:UID:GID:comment:home:shell. Accounts with shell /usr/sbin/nologin are service accounts that cannot log in interactively — these should never be modified by attackers.
  • 4
    Navigate to /var/log then run ls -la. List what log files exist. On an EC2 instance, these contain your evidence trail during an incident.
  • 5
    If auth.log exists, run: grep "Failed" /var/log/auth.log | head -20. Even on a fresh system you may see automated bots attempting SSH logins within minutes of the server going online — this is real.
  • 6
    Run find /etc -name "*.conf" 2>/dev/null. Count how many config files exist. Each one is a potential security misconfiguration to audit. The 2>/dev/null suppresses permission errors.
  • 7
    Write in notes: Where are system logs? Where are user accounts listed? What does the /tmp directory significance for security investigations?

File Permissions, Ownership & User Management
⏱ 1.5 hrs📖 Lesson + chmod Deep Dive🔬 Permissions Lab
Lesson

Linux Permissions — OS-Level Least Privilege

Linux permissions enforce least privilege at the operating system level — the same principle you implement in IAM at the cloud level. Every file and directory has an owner, a group, and permissions defining who can read, write, or execute it. Misconfigured permissions are a direct attack path.

Reading the Permission String

When you run ls -la, each line shows: -rwxr-xr-- 1 ankit developers 4096 Feb 27 script.sh

First character: file type (- = file, d = directory, l = symlink) Next 3: Owner permissions → rwx = read, write, execute Next 3: Group permissions → r-x = read, no write, execute Last 3: Others permissions → r-- = read only "ankit" = file owner · "developers" = group owner
Octal Permission Values — The Numbers Behind chmod

Read=4, Write=2, Execute=1. Add for combinations:

7 = rwx (4+2+1) — full access 6 = rw- (4+2) — read and write 5 = r-x (4+1) — read and execute 4 = r-- (4) — read only 0 = --- — no permissions chmod 755 = Owner:rwx(7) Group:r-x(5) Others:r-x(5) → standard for scripts chmod 644 = Owner:rw-(6) Group:r--(4) Others:r--(4) → standard for config files chmod 600 = Owner:rw-(6) Group:---(0) Others:---(0) → SSH private keys ONLY
# Change permissions chmod 755 script.sh # owner=rwx, group+others=r-x chmod 600 private.key # owner rw only — SSH key standard chmod +x script.sh # add execute for all users chmod -R 755 /var/www # recursive — apply to directory+contents # Change ownership chown ankit file.txt # change owner chown ankit:devs file.txt # change owner and group # User management (requires sudo/root) sudo adduser newuser # create new user sudo usermod -aG sudo newuser # add to sudo group sudo passwd newuser # set password sudo userdel -r olduser # delete user + home directory sudo -l # what sudo commands can I run?
⚠️ SSH Key Permissions Are Enforced by SSH Itself

SSH private keys MUST have permissions 600 or 400. If your key file is 644 (world-readable), SSH will refuse to use it and print "WARNING: UNPROTECTED PRIVATE KEY FILE!" — this is a security enforcement built into SSH. A private key readable by others is considered compromised. Always run chmod 600 ~/.ssh/id_rsa immediately after creating or copying SSH keys.

Processes, Services & Package Management
⏱ 1.5 hrs📖 Lesson + Commands🔬 Process Investigation Lab
Lesson

What is Running on This Server Right Now?

During incident response, one of the first questions is: "What processes are running that shouldn't be?" A cryptominer consuming 100% CPU, a backdoor listening on port 4444, a data exfiltration script copying files somewhere — all of these show up in the process list if you know where to look.

# View running processes ps aux # all processes, all users, detailed ps aux | grep nginx # filter for specific process name top # live monitor, sorted by CPU (q to quit) # Network connections — find backdoors ss -tulpn # all listening ports + owning process ss -tp # established connections # Unexpected listening port = potential backdoor # Kill processes kill 1234 # graceful stop (process ID 1234) kill -9 1234 # force kill — cannot be ignored # Service management (systemd) systemctl status nginx # is it running? last logs? systemctl start nginx # start now systemctl stop nginx # stop now systemctl enable nginx # auto-start on boot systemctl disable nginx # don't start on boot systemctl list-units --type=service --state=running # all running services # Package management — patching = security sudo apt update && sudo apt upgrade -y # Ubuntu: update all packages sudo yum update -y # Amazon Linux/CentOS: update all sudo apt install fail2ban # install a security tool
⚠️ Patch Management = Security, Not Just Maintenance

The #1 cause of server compromises is unpatched software. When a CVE (Common Vulnerability and Exposure) is published, automated scanners begin testing every internet-facing server within hours. In AWS, use Systems Manager Patch Manager to automate patching across all EC2 instances on a schedule. Establishing an automated patch management policy is often one of the first deliverables for a Cloud Security Engineer at a new company.

Hands-On Lab
Lab 4
Investigate Running Processes & Network Connections
  • 1
    Run ps aux. Find the columns: USER (who runs it), PID (process ID), %CPU, %MEM, COMMAND (what is it). Look for processes running as root — these are highest-risk targets if compromised.
  • 2
    Filter for specific processes: ps aux | grep ssh. Find the sshd process — this is the SSH daemon accepting incoming connections. Note its PID. Note it runs as root initially (required to bind port 22), then drops privileges after auth.
  • 3
    Run ss -tulpn. Look at each listening port. The columns: Netid (protocol), State (LISTEN), Local Address:Port, Process. Any port you don't recognize is worth investigating — cross-reference against what services you know should be running.
  • 4
    Run top. Watch the live process list. Press P to sort by CPU. Press M to sort by memory. Press q to quit. A process unexpectedly consuming 100% CPU with an unusual name is suspicious — potentially a cryptominer.
  • 5
    Run systemctl list-units --type=service --state=running. How many services are running? On a minimal secure server, this list should be as short as possible — every running service is an attack surface. "Less is more" in server security.
  • 6
    Write in notes: What command shows listening ports AND the process using each? If you saw an unknown process listening on port 4444, what steps would you take? (Hint: identify the process, check its file, check CloudTrail, isolate if needed.)

SSH — Secure Remote Access to Cloud Servers
⏱ 1.5 hrs📖 Lesson + Key Pairs🔬 SSH Key Generation Lab
Lesson

SSH — How You Access Every Cloud Server

SSH (Secure Shell) is how you securely connect to and control remote Linux servers. Every EC2 instance you access, every security investigation on a live system, every deployment — SSH is how you get there. Understanding SSH at a deep level is non-negotiable for Cloud Security Engineers and Architects.

How SSH Key Pair Authentication Works

SSH supports two auth methods: password (weaker — brute-forceable) and key pair (stronger — what you always use in cloud environments).

Key pair auth uses asymmetric cryptography:
Private key — lives on YOUR computer only. Never share it. Mathematically paired with the public key.
Public key — stored on the server in ~/.ssh/authorized_keys. Safe to share.

When you connect: SSH uses your private key to solve a mathematical challenge set by the server. The server verifies your answer using the stored public key. If correct — you're in. The private key is never transmitted.

You → Server: "I want to connect"
Server → You: "Here's a challenge (encrypted with your public key)"
You → Server: "Here's the solution (only possible with the matching private key)"
Server: "Correct — access granted" (private key never left your machine)
# Generate a key pair (use ed25519 — modern and secure) ssh-keygen -t ed25519 -C "ankit-work-2025" # Creates: ~/.ssh/id_ed25519 (private) + ~/.ssh/id_ed25519.pub (public) # Set a passphrase for extra security # Correct permissions (SSH enforces these) chmod 700 ~/.ssh # directory: owner only chmod 600 ~/.ssh/id_ed25519 # private key: owner read/write only chmod 644 ~/.ssh/id_ed25519.pub# public key: readable by all (fine) # Connect to EC2 instance ssh -i ~/.ssh/mykey.pem ec2-user@52.60.123.45 # ec2-user = Amazon Linux default user # ubuntu = Ubuntu EC2 default user # Copy files securely scp -i mykey.pem localfile.txt ec2-user@52.60.x.x:~/ # to server scp -i mykey.pem ec2-user@52.60.x.x:~/log.txt ./ # from server
SSH Hardening — /etc/ssh/sshd_config Best Practices

These settings in the SSH server config dramatically reduce your attack surface. After changing sshd_config, run sudo systemctl restart sshd:

PermitRootLogin no # Never allow direct root SSH — forces privilege escalation step PasswordAuthentication no # Key pairs only — eliminates brute force entirely MaxAuthTries 3 # Lock out after 3 failed attempts Port 22022 # Non-standard port — reduces automated scan noise AllowUsers ankit deploy # Whitelist: only these accounts can SSH in ClientAliveInterval 300 # Disconnect idle sessions after 5 minutes
⚠️ Test Your Key Before Disabling Passwords

If you set PasswordAuthentication no without first verifying your SSH key works, you'll permanently lock yourself out of the server. Always test SSH key access in a separate terminal session BEFORE disabling password auth. In AWS, if you lose SSH access, AWS Systems Manager Session Manager provides browser-based shell access as a fallback — set this up from day one on every EC2 instance.

    The private key never leaves your machine — authentication uses mathematical proof; immune to brute force, phishing, and credential stuffing attacks
  • C
    Key-based auth is easier to manage than passwords at scale
  • D
    Strong passwords are equally secure — the difference is minimal in practice
✓ Correct. Key-based auth eliminates three major attack categories: (1) Brute force — there's no password to guess (private key = 256 bits of randomness, mathematically infeasible to brute force); (2) Phishing — there's no "type your password" interaction to trick; (3) Credential stuffing — your key can't be in a leaked credential database. Bots continuously scan the internet for SSH servers — PasswordAuthentication no makes them completely irrelevant.

Scripts Beat Manual Commands Every Time

A Bash script is a text file containing Linux commands executed in sequence. When you need to run the same security checks on 50 servers, or audit configurations daily, or respond to an incident with a consistent process — you don't do it manually. You write the script once and run it everywhere, automatically.

As a Cloud Security Engineer, you'll write scripts to: audit security configurations across fleets, parse logs for suspicious patterns, automate incident collection steps, and deploy controls at scale.

#!/bin/bash # Shebang — tells OS to use bash to run this file # Make executable with: chmod +x script.sh # Run with: ./script.sh or bash script.sh # Variables SERVER_NAME="equiton-prod-01" LOG_DIR="/var/log" echo "Auditing: $SERVER_NAME at $(date)" # If/else conditionals if [ -f "/etc/passwd" ]; then echo "User accounts file found" else echo "WARNING: /etc/passwd missing!" fi # For loops — do something for each item for LOG in /var/log/*.log; do ERRORS=$(grep -c "ERROR" "$LOG" 2>/dev/null) echo "$LOG: $ERRORS errors" done # Pipe commands together — real forensics power # Top 10 IPs brute-forcing your SSH server: grep "Failed password" /var/log/auth.log \ | awk '{print $11}' \ | sort | uniq -c | sort -rn | head -10 # Output: "count IP_address" — highest counts = active attackers

    echo "========================================"
    echo "SECURITY AUDIT: $(hostname) — $(date)"
    echo "========================================"
    echo ""
    echo "[1] FAILED SSH LOGIN ATTEMPTS (last 50):"

    echo ""
    echo "[2] TOP BRUTE-FORCE SOURCE IPs:"

    echo "[3] LISTENING PORTS (potential backdoors):"

    echo ""

Question 1 of 3
What does the #!/bin/bash line at the top of a script do?
  • A
    It's a comment line — purely documentation, no functional effect
  • B
    It's the shebang — tells the OS to use /bin/bash to interpret this script when it's executed directly
  • C
    It imports the bash standard library so commands like echo are available
  • D
    It sets the script to run as root automatically
✓ Correct. The shebang (#! followed by an interpreter path) tells the kernel which program to use to execute the file when run directly (./script.sh). Without it, the OS may not know how to execute the file. You also need chmod +x to set execute permission. Without both the shebang AND execute permission, the script can only be run as: bash script.sh (explicitly invoking bash).
Question 2 of 3
This command: grep "Failed password" /var/log/auth.log | awk '{print $11}' | sort | uniq -c | sort -rn | head -10 — what does it produce?
  • A
    The 10 most recent failed login attempts with timestamps
  • B
    The top 10 source IP addresses with the most failed SSH attempts — a brute-force attack detection report
  • C
    The 10 users with the most failed login attempts
  • D
    All failed login attempts from the last 10 hours
✓ Correct. Breaking down the pipe chain: grep extracts lines with "Failed password" → awk '{print $11}' extracts the 11th space-separated field (the source IP) → sort alphabetically groups duplicates → uniq -c counts each unique IP → sort -rn sorts by count descending → head -10 shows the top 10. Output looks like: "4821 203.0.113.5" — that IP tried 4,821 times. This is active brute force. Block it at the NACL immediately.

Question 1 of 3
You find a file in /tmp named "update_agent.sh" owned by "www-data" (the web server user) with permissions 777. What does this suggest?
  • A
    A legitimate system update script — /tmp is the standard location for update files
  • B
    High-confidence compromise indicator — a web server process wrote an executable script to /tmp, a classic web shell exploitation pattern used for privilege escalation
  • C
    Normal web server maintenance — Apache and Nginx regularly write scripts to /tmp
  • D
    Low risk — /tmp files are auto-deleted on reboot so there's no persistence
✓ Correct. This is the classic web application compromise pattern: attacker exploits a vulnerability in the web app → uses the www-data process to write a malicious script to /tmp (world-writable, so any process can write there) → sets permissions 777 (so any user can execute it) → uses it to escalate privileges. Immediate response: don't execute it, copy it for analysis, isolate the server, check web access logs for the exploit attempt, and begin full incident response. The file being in /tmp does NOT mean it's harmless.
