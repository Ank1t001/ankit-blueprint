AWS IAM — Identity & Access Management

Day 1 of 7
IAM Fundamentals — Why Access Management Is Security
⏱ 1.5 hrs📖 Concepts + AWS Console🔬 Root vs IAM User Lab
Lesson

The Most Misunderstood AWS Service Is Also the Most Critical

Most AWS beginners focus on EC2, S3, and databases — the visible infrastructure. But IAM is the layer that controls who can touch any of it. A perfectly architected VPC with tight security groups means nothing if an over-privileged IAM user gets compromised and deletes your databases or exfiltrates your data. The biggest cloud breaches in history weren't network failures — they were IAM failures.

IAM (Identity and Access Management) answers one question: "Is this principal allowed to perform this action on this resource?" Every API call to AWS — creating an EC2 instance, reading an S3 object, modifying a security group — goes through IAM. There are no exceptions.

The Four IAM Building Blocks

Identities (Who?): Users, Groups, Roles — the entities that make requests
Policies (What?): JSON documents that define allowed/denied actions
Resources (On what?): The AWS services and objects being accessed
Conditions (When/how?): Extra constraints like MFA required, specific IP, time of day

Request: "Can user 'ankit-dev' perform 's3:GetObject' on 'arn:aws:s3:::equiton-data/investor-reports/*' ?" IAM evaluates: identity → attached policies → conditions → ALLOW or DENY
The AWS Root Account — Your Most Dangerous Credential

When you create an AWS account, you get a root account — the email + password used to sign up. The root account has unrestricted access to everything in the account, including billing, account closure, and IAM itself. There are only a handful of things only root can do (change account email, close the account, restore IAM access). For everything else, you should use IAM users or roles.

Root credentials are so dangerous that AWS explicitly recommends: create the account, enable MFA on root, create an admin IAM user, then lock root credentials in a safe and never use them again.

Root CAN: close the AWS account, change account email, restore deleted IAM admin access Root SHOULD NOT: be used for daily operations, have access keys, be shared with anyone
🏢 Equiton Analogy

Think of your AWS account like Equiton's office building. The root account is the building owner who has a master key, can change the locks, and can fire everyone — you'd keep that key in a vault. IAM users are like individual employee keycards — each person gets only the access their job requires. IAM roles are like temporary visitor badges with specific access — "you can be in the server room for the next hour." IAM policies are the rulebook defining what each badge can open.

IAM Users, Groups & Roles — The Three Identities
⏱ 1.5 hrs📖 Lesson + Design Patterns🔬 Create User & Group Lab
Lesson

Users, Groups, and Roles — When to Use Each

IAM has three types of identity. Choosing the wrong one is one of the most common IAM mistakes — especially using users where roles should be used. Understanding the difference is a Cloud Security Architect competency.

IAM Users — Long-Term Human Identities

An IAM user is a permanent identity with a username, password (for Console access), and optionally access keys (for programmatic API access). Users are appropriate for human beings who need long-term access to the AWS Console or CLI.

Each user should have only the minimum permissions needed — if a developer only works with EC2 and S3, they get an EC2+S3 policy, not AdministratorAccess.

ankit-admin → AdministratorAccess (emergency use only, MFA required) dev-priya → EC2ReadOnly + S3FullAccess (for specific projects) ops-carlos → CloudWatchReadOnly + SSMFullAccess
IAM Groups — Apply Policies to Teams, Not Individuals

Groups are collections of users. You attach policies to the GROUP, not individual users. When a new developer joins, add them to the "Developers" group — they automatically inherit the right permissions. When they leave, remove them from the group. Never attach policies directly to individual users when groups are available. This makes access management scalable and auditable.

Group: Developers → AmazonEC2FullAccess, AmazonS3ReadOnly Group: SecurityTeam → SecurityAudit, AWSSecurityHubReadOnlyAccess, CloudTrailReadOnly Group: Finance → AWSBillingReadOnlyAccess, CostExplorerReadOnly
IAM Roles — Temporary Credentials for Services & Automation

Roles are the most important and most underused IAM concept. A role has no permanent credentials (no password, no access keys). Instead, it's assumed temporarily by a trusted entity — an AWS service, another AWS account, or a user needing elevated access.

The golden rule: Never create IAM access keys for an EC2 instance, Lambda, or any AWS service. Give the resource an IAM role. The role provides temporary, automatically-rotated credentials that never leave the AWS environment.

EC2 instance → assumes role "WebServerRole" → can read S3 objects (no access keys in code!) Lambda function → assumes role "ETLRole" → can read DynamoDB + write S3 Developer → assumes role "ProdReadOnlyRole" → temporary access to prod (1-hour session)
⚠️ The #1 IAM Anti-Pattern: Access Keys on EC2

Never put IAM access keys (AWS_ACCESS_KEY_ID / AWS_SECRET_ACCESS_KEY) in EC2 user data, environment variables, config files, or — worst of all — code committed to GitHub. If the instance is compromised or the code repo is public, the keys are immediately stolen. Use IAM roles instead: the instance automatically gets temporary credentials through the instance metadata service. No keys to store, no keys to rotate, no keys to leak.

IAM Policies & the Principle of Least Privilege
⏱ 1.5 hrs📖 Policy JSON Deep Dive🔬 Write & Test Custom Policies
Lesson

IAM Policies — The JSON That Controls Everything

IAM policies are JSON documents that define permissions. Mastering policy structure is what separates engineers who "configure access" from architects who "design access systems." Every permission in AWS is ultimately defined by policy JSON — understanding it gives you full control.

Policy Structure — The Four Elements

Every policy statement has four components: Effect, Action, Resource, and optionally Condition.

{ "Version": "2012-10-17", ← Always use this version "Statement": [ { "Effect": "Allow", ← Allow or Deny "Action": [ "s3:GetObject", ← Specific API action "s3:ListBucket" ], "Resource": [ "arn:aws:s3:::equiton-reports", ← Specific bucket "arn:aws:s3:::equiton-reports/*" ← Objects inside it ], "Condition": { ← Optional extra constraints "Bool": { "aws:MultiFactorAuthPresent": "true" ← Require MFA } } } ] }
Policy Evaluation Logic — The Deny/Allow Hierarchy

AWS evaluates policies in a specific order. An explicit Deny always wins — even over an explicit Allow. If no policy explicitly allows an action, it is denied by default (implicit deny).

Order of evaluation: (1) Check for explicit Deny → if found, DENY immediately. (2) Check for explicit Allow → if found, ALLOW. (3) If neither, DENY (implicit default).

Scenario: User has "S3FullAccess" policy attached (Allows everything) AND a separate policy with "Deny s3:DeleteObject" on the data bucket Result: The explicit Deny wins. The user CANNOT delete objects from that bucket, even though S3FullAccess would normally allow it.

Principle of Least Privilege — The Core Security Principle

Least privilege means: give every identity exactly the access it needs to do its job, and nothing more. Not "roughly what they need." Not "this AWS managed policy is close enough." Exactly what they need — specific actions, specific resources, specific conditions.

In practice, this means writing resource ARNs to specific buckets and prefixes, not using * as a wildcard resource. It means allowing s3:GetObject on investor reports, not s3:* on all S3. It means adding MFA conditions to sensitive operations.

💡 AWS IAM Access Analyzer — Your Least Privilege Assistant

IAM Access Analyzer reviews CloudTrail logs and tells you: "This user has permission to do X, Y, Z — but has only ever actually done X. Consider removing Y and Z." Use it in every account to right-size permissions over time. It's free, built into IAM, and directly addresses the lazy-admin problem of "just give them admin access and we'll tighten later" — which never actually happens.

MFA, Access Keys & Credential Security
⏱ 1.5 hrs📖 Lesson + Credential Hygiene🔬 MFA & Key Rotation Lab
Lesson

The Credential Attack Surface — Every Way Attackers Get In Via IAM

IAM compromises happen in predictable ways: stolen passwords, leaked access keys, phished users, or unrotated credentials. Each has a specific defense. As a Cloud Security Engineer, your job is to make each attack vector as difficult as possible through credential hygiene.

Multi-Factor Authentication (MFA) — The Most Important Control

MFA requires two factors to authenticate: something you know (password) + something you have (TOTP code from authenticator app, hardware security key). Even if an attacker steals a user's password, they cannot log in without the second factor.

MFA Types in AWS: Virtual MFA: Google Authenticator, Authy app (free, good for most users) Hardware MFA: YubiKey security key (best for privileged/admin accounts) SMS MFA: Not recommended (SIM-swap attacks can intercept SMS codes) Enforcement: Create an IAM policy with Deny on all actions if MFA not present. Attach to all users. Even if they have Admin permissions — no MFA = no access.
IAM Access Keys — For Programmatic Access, Handle with Care

Access keys (Access Key ID + Secret Access Key) provide programmatic access to AWS via CLI, SDK, or API. They are long-lived credentials with no expiry by default — making them high-value targets for attackers. Every access key in your account is a potential breach vector if not managed carefully.

Access Key Best Practices: ✓ Never create access keys for the root account ✓ Rotate access keys every 90 days (automated with AWS Config rule) ✓ Delete keys that haven't been used in 90 days (Credential Report shows last used date) ✓ Use IAM roles instead of access keys wherever possible ✓ Use AWS Secrets Manager or environment variables — never hardcode in code ✗ Never commit keys to GitHub (automated scanners find them in seconds)

The IAM Credential Report — Your Access Audit Tool

The Credential Report is a CSV file AWS generates showing every IAM user in your account with: when their password was last used, whether MFA is enabled, when their access keys were last rotated, and whether keys have been used recently. This is your baseline for access hygiene audits — run it monthly in any production environment.

💡 The MFA Enforcement Policy — High-Value Security Control

Create a policy called "RequireMFAForAllActions" with a Deny on all actions when MFA is not present. Attach it to every IAM user. This single policy means stolen passwords are useless — attackers need the physical authenticator too. In OSFI B-13 and financial services compliance, MFA enforcement for privileged access is typically a mandatory control. This is a policy you'd implement on day one as a Cloud Security Engineer at any financial firm.

IAM Roles for AWS Services — How the Cloud Talks to Itself
⏱ 1.5 hrs📖 Service Roles Deep Dive🔬 Attach Role to EC2 Lab
Lesson

Service-to-Service Access — The Most Common Role Use Case

In modern cloud architectures, AWS services constantly talk to each other: EC2 reads from S3, Lambda writes to DynamoDB, CodePipeline deploys to ECS. Each of these service-to-service calls needs permissions — and every one of them should use IAM roles, not access keys. This is how AWS was designed to work.

Common Service Role Patterns — Financial Services Context
EC2 Web Server → Role: ReadFromS3, WriteToCloudWatchLogs Lambda ETL Function → Role: ReadDynamoDB, WriteS3, PublishSNS ECS Container → Role: ReadSecretsManager, WriteRDS connections CodePipeline CI/CD → Role: AssumeRole in target accounts, DeployEC2, UpdateECS Backup service → Role: CreateSnapshots, ReadAllEC2, RestrictedWriteS3
How Instance Profiles Work — No Key Storage Needed

When you attach a role to an EC2 instance (via an instance profile), the EC2 metadata service at 169.254.169.254 provides temporary credentials to any process on the instance. These credentials automatically rotate every hour. Your application code calls the AWS SDK — it automatically fetches credentials from the metadata service. No keys to store, no expiry to manage.

Application code (Python): import boto3 s3 = boto3.client('s3') # SDK auto-fetches credentials from metadata s3.get_object(Bucket='equiton-reports', Key='investor-q3.pdf') # No credentials in code — the instance role provides them automatically
⚠️ Instance Metadata SSRF — A Real Attack Pattern

The instance metadata endpoint at 169.254.169.254 is accessible from any process on the EC2 instance — including web applications. A Server-Side Request Forgery (SSRF) vulnerability in your web app lets attackers trick it into fetching the metadata endpoint, which returns the temporary IAM role credentials. They can then use those credentials from outside AWS. Defense: use IMDSv2 (requires a session token, blocking simple SSRF) — enforce it in EC2 launch settings and via AWS Config rules.

    IAM → Roles → Create role. Trusted entity: AWS service → EC2. This creates a role that EC2 instances can assume. Click Next.
  • 2
    Search for and attach: AmazonS3ReadOnlyAccess AND CloudWatchAgentServerPolicy. These two together represent a common web server role: can read application config from S3, can send logs and metrics to CloudWatch.

A Lambda function processes investor transactions and needs to read from DynamoDB and write results to S3. How should it be granted these permissions?
  • A

  • B

  • C

  • D
    Make both the DynamoDB table and S3 bucket public — Lambda can then access without credentials
✓ Correct. Every Lambda function has an execution role — an IAM role that Lambda assumes when the function runs. The function's code automatically receives temporary credentials from this role. No keys stored anywhere, no rotation needed, no exposure risk. If the function is compromised, the blast radius is limited to exactly what the execution role permits — this is least privilege in action for serverless architectures.
Question 2 of 3
What advantage do IAM role temporary credentials have over static IAM access keys for EC2 applications?
  • A
    Temporary credentials grant more permissions than static access keys
  • B
    They automatically rotate every hour, never need to be stored in code or files, and are automatically revoked if the role is removed — eliminating key management overhead and leakage risk
  • C
    Temporary credentials cannot be stolen — they are only accessible to AWS services internally
  • D
    They work offline without internet connectivity, unlike static keys
✓ Correct. The security advantages of temporary credentials from roles: (1) Auto-rotate every hour — a stolen credential has a 1-hour maximum useful window; (2) Never stored anywhere — no config files, no environment variables, no git commits to expose; (3) Instantly revokable — modify or remove the role and all instances immediately lose access; (4) No rotation schedule — there's nothing to forget to rotate. This is the future of cloud credentials: ephemeral, automatically managed, zero operational overhead.
Question 3 of 3
A web app running on EC2 with an IAM role has an SSRF vulnerability. An attacker exploits it to fetch http://169.254.169.254/latest/meta-data/iam/security-credentials/MyRole. What do they obtain?
  • A
    Nothing — the metadata endpoint is not accessible from web applications
  • B
    The instance's temporary IAM credentials (AccessKeyId, SecretAccessKey, Token) — usable from outside AWS to perform any action the role permits
  • C
    Only the role name — not the actual credentials, which are encrypted
  • D
    The AWS account ID only — a low-severity information disclosure
✓ Correct. SSRF against the EC2 metadata endpoint is a high-severity attack that has compromised multiple major companies (including the infamous Capital One breach). The metadata endpoint returns actual usable credentials with no authentication. IMDSv2 defense: requires a PUT request to get a session token before any GET — SSRF attacks typically can't make PUT requests through the vulnerability. Always enforce IMDSv2 via EC2 launch templates and AWS Config rules. Never assume "our app doesn't have SSRF" — it needs to be tested.

Designing IAM at Scale — From Engineer to Architect Thinking

IAM best practices for a single account are relatively straightforward. But Cloud Security Architects design IAM for entire organizations — multiple accounts, hundreds of users, dozens of services, cross-account access patterns, and compliance requirements. This day bridges the CCP content with how you'll actually design access systems in your target role.

AWS Organizations & Service Control Policies (SCPs)

Large organizations use AWS Organizations to manage multiple AWS accounts. An organization might have: a management account, a security account, a logging account, and per-environment accounts (dev, staging, prod). Service Control Policies (SCPs) are applied at the organizational unit level and act as guardrails — they limit what IAM policies in member accounts can allow.

Key distinction: SCPs don't grant permissions. They restrict the maximum permissions available. Even an AdministratorAccess policy in a member account is constrained by the SCP applied to that account's OU.

SCP Example — Restrict to Canadian Regions Only: { "Effect": "Deny", "Action": "*", "Resource": "*", "Condition": { "StringNotEquals": { "aws:RequestedRegion": ["ca-central-1", "ca-west-1"] } } } Effect: No one in this account can provision ANY service outside Canada — even the account admin. PIPEDA compliance enforced at the guardrail level, not policy-by-policy.
IAM Best Practices Checklist — The Architect's Baseline
✓ Root account: MFA enabled, no access keys, hardware security key for privileged users ✓ All IAM users: MFA enforced via policy (Deny if aws:MultiFactorAuthPresent = false) ✓ Users in groups: never attach policies directly to individual users ✓ Roles for services: no access keys on EC2, Lambda, ECS, or any AWS service ✓ Least privilege: custom policies with specific ARNs, not AWS managed FullAccess policies ✓ Access key hygiene: rotate every 90 days, delete if unused 90 days (automate with Config) ✓ Credential report: run monthly, review stale credentials ✓ CloudTrail enabled: all IAM API calls logged in all regions ✓ IAM Access Analyzer: continuously identify unused permissions ✓ Permission boundaries: limit what delegated admins can grant to others
💡 This Is What a Cloud Security Architect Delivers on Day 30

When you get to your Cloud Security Architect role, one of your first deliverables will be an IAM security baseline document: root account controls, MFA policy, SCP structure, role naming conventions, group taxonomy, access key lifecycle policy, and a quarterly credential review process. Every item in today's best practices checklist becomes a policy document section. The technical knowledge from this week translates directly into that deliverable.

Question 1 of 3
A Canadian financial services company needs to ensure all AWS data stays in Canadian regions (ca-central-1, ca-west-1) for PIPEDA compliance. Which control enforces this at the organizational level?
  • A
    IAM policies attached to all users denying cross-region access
  • B
    A Service Control Policy (SCP) denying all actions in non-Canadian regions, applied at the AWS Organizations OU level
  • C
    S3 bucket policies restricting upload to Canadian IPs only
  • D
    VPC configuration limiting EC2 launches to Canadian availability zones
✓ Correct. SCPs are the right tool for organizational-level guardrails. Unlike IAM policies (which can be modified by account admins), SCPs are applied from the management account and constrain all accounts in the OU — even account administrators can't override them. This makes SCPs the correct enforcement mechanism for regulatory requirements like PIPEDA data residency. IAM policies in individual accounts would require trusting every account admin to configure them correctly — SCPs remove that dependency.
Question 2 of 3
IAM Access Analyzer reports that an S3 bucket has a policy allowing access from any AWS account (principal: "*"). What does this mean?
  • A
    The bucket is protected — only other AWS accounts can access it, not the general public
  • B
    The bucket is publicly accessible to any AWS account globally — including anonymous — which is likely a misconfiguration and a data exposure risk
  • C
    Only IAM users in the current account can access the bucket
  • D
    The finding is informational — Access Analyzer flags all cross-account access as suspicious
✓ Correct. A bucket policy with Principal "*" and Effect "Allow" makes the bucket publicly readable by anyone with an AWS account (or in some configurations, anyone at all). This is the configuration that caused the Capital One breach and countless others — sensitive data in an S3 bucket with an overly permissive bucket policy. IAM Access Analyzer exists specifically to surface these misconfigurations. Any active finding should be investigated and remediated unless the bucket is intentionally public (e.g., a static website).
Question 3 of 3
Which IAM configuration represents the most secure setup for a financial services AWS account with 5 developers?
  • A
    All 5 developers share one IAM user for simplicity — easier to manage access keys
  • B
    Each developer has their own IAM user with AdministratorAccess — ensures they can always do their job
  • C
    Each developer has their own IAM user with MFA enforced, added to role-specific groups (Frontend, Backend, DevOps), with least-privilege group policies and CloudTrail logging all API calls
  • D

    • ✓ Correct. Option C implements every IAM best practice: individual accounts (audit trail per person — you know who did what), MFA (phishing-resistant authentication), groups with least-privilege policies (right access for each role, scalable to onboard/offboard), and CloudTrail (immutable log of every API call).     Question 1 of 3
    A security incident investigation reveals that an attacker used IAM credentials to enumerate all S3 buckets (s3:ListAllMyBuckets), then downloaded data from three buckets. The credentials belonged to an IAM user used by an EC2 instance. What was the root cause failure?
    • A
      The S3 buckets were not encrypted — encrypted buckets cannot be accessed this way

      The VPC security groups were too permissive, allowing the attacker network access

      MFA was not enabled — the attacker guessed the password
    ✓ Correct. This is the IAM user-on-EC2 anti-pattern in action. Access keys stored on EC2 can be leaked through: application vulnerabilities, instance compromise, accidental git commit, misconfigured metadata service, or environment variable exposure. Once leaked, they're usable from anywhere globally. The fix: replace the IAM user with an IAM role attached to the EC2 instance. Role credentials rotate hourly, aren't stored anywhere accessible, and can't be used from outside the instance without additional effort.
    Question 2 of 3
    An AWS Organization has an SCP that denies all actions outside ca-central-1. An account admin with AdministratorAccess tries to launch an EC2 instance in us-east-1. What happens?
    • A
      The launch succeeds — AdministratorAccess overrides all restrictions including SCPs
    • B
      The launch fails — SCPs constrain the maximum permissions available in the account; even AdministratorAccess cannot exceed SCP boundaries
    • C
      The launch succeeds but triggers a compliance alert in Security Hub
    • D
      The launch succeeds — only non-admin users are constrained by SCPs
    ✓ Correct. SCPs define the ceiling — the maximum permissions available in an account. AdministratorAccess grants all IAM permissions within that ceiling, but cannot exceed it. If the SCP denies us-east-1, even the account's root user cannot launch there (SCPs apply to all principals including root in member accounts). This is exactly why SCPs are the right tool for hard compliance requirements like data residency — they're enforced at the organization level, above the account level, making them resistant to insider bypass.
