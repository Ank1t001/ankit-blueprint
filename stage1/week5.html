EC2 Fundamentals — Virtual Servers in the Cloud
⏱ 1.5 hrs📖 Lesson + Instance Types🔬 Launch Your First EC2
Lesson

EC2 — Elastic Compute Cloud

Amazon EC2 provides on-demand virtual servers — you choose the operating system, memory, CPU, and storage, and AWS provides the physical hardware underneath. Every workload that needs a running process — web servers, application servers, databases, security tools, batch jobs — lives on EC2 or a service built on top of it.

From a security standpoint, EC2 is your primary attack surface: misconfigured security groups expose open ports, overly permissive IAM roles enable lateral movement, unpatched AMIs introduce known vulnerabilities, and public-facing instances are scanned by bots within minutes of launch. Understanding EC2 security is foundational to your Cloud Security Architect role.

The EC2 Launch Decision Tree

Every time you provision an EC2 instance, you make five key decisions that shape its security posture:

  • 1
    AMI (Amazon Machine Image): The OS blueprint — which operating system, which version, pre-installed software. Determines base vulnerability footprint.
  • 2
    Instance Type: Hardware specs — CPU family, memory, network. t3.micro (free tier) → c5.xlarge (compute) → r5.2xlarge (memory). Affects cost, not security directly.
  • 3
    Network: Which VPC, which subnet (public vs private), auto-assign public IP. Determines internet reachability.
  • 4
    IAM Role: Which AWS permissions does this instance need? Use least privilege. Critical security decision.
  • 5
    Security Group: Which inbound/outbound ports are open? Your first line of defence.
Instance Types — The Family Naming System

EC2 instance type names follow a pattern: family + generation + size

t3.micro → t=burstable (general workloads) · gen 3 · micro size → 1 vCPU, 1GB RAM m5.large → m=general purpose · gen 5 · large → 2 vCPU, 8GB RAM c5.xlarge → c=compute optimized · gen 5 · xlarge → 4 vCPU, 8GB RAM r5.4xlarge → r=memory optimized · gen 5 · 4xlarge → 16 vCPU, 128GB RAM p3.2xlarge → p=GPU (ML/AI workloads) · gen 3 · 2xlarge → 8 vCPU + V100 GPU
Purchasing Options — Pay Only for What You Use

EC2 has 4 pricing models with very different cost profiles:

On-Demand: Full price, start/stop anytime — good for dev/test, unpredictable workloads Reserved: 1–3 year commitment, up to 72% discount — production steady-state workloads Spot: Bid for unused capacity, up to 90% discount — can be terminated with 2min notice Savings Plan: Commit to $ spend/hr, flexible across instance types — best of both worlds
🏢 Marketing Analogy

Running digital ads? On-Demand pricing is like paying full CPM rate with no commitment — flexible but expensive at scale. Reserved instances are like negotiating an annual media buy with a publisher — big discount for volume commitment. Spot instances are like buying remnant inventory last-minute at steep discount — great savings but no guaranteed delivery.

EC2 Security — AMIs, Security Groups & Instance Metadata
⏱ 1.5 hrs📖 Security Deep Dive🔬 Harden Your Instance
Lesson

AMI Security — Your Security Starts at the Image

An AMI (Amazon Machine Image) is a complete snapshot of an EC2 instance — operating system, installed software, configuration, and data. When you launch from an AMI, you clone that state. A compromised or vulnerable AMI means every instance launched from it is immediately vulnerable.

Three AMI sources with different trust levels:

AMI Trust Hierarchy

AWS-Managed AMIs (Amazon Linux, Ubuntu official) — highest trust. AWS patches these, maintains security advisories, and you can verify them by AWS account ID. Always prefer these.

AWS Marketplace AMIs — vendor-published, AWS-verified. Check publisher reputation, last update date, and CVE track record before use. Some marketplace AMIs haven't been updated in years.

Community AMIs / Custom AMIs — lowest trust by default. Treat as potentially compromised until verified. Never use community AMIs for production without security scanning (Amazon Inspector).

AWS Console → EC2 → AMIs → Public Images → Filter by owner: amazon Verify AMI owner account ID for AWS-published images: 137112412989 (Amazon Linux) Always check: creation date, OS version, last security update

The EC2 Instance Metadata Service (IMDS) — The Most Misunderstood Attack Surface

Every EC2 instance has access to a special internal endpoint at http://169.254.169.254 — the Instance Metadata Service (IMDS). This endpoint provides the instance with its own metadata: region, instance ID, public IP, and critically, the temporary credentials of the attached IAM role.

This became the attack vector in Capital One's 2019 breach: a misconfigured WAF allowed SSRF (Server-Side Request Forgery), which an attacker used to make the web server request its own IAM credentials from IMDS, then used those credentials to access 100 million customer records in S3.

⚠️ IMDSv1 vs IMDSv2 — The Version That Matters

IMDSv1 (legacy): Simple HTTP GET to 169.254.169.254/latest/meta-data/iam/security-credentials/ returns IAM credentials with no authentication. Any process on the instance (including a compromised web app via SSRF) can retrieve them.

IMDSv2 (required): Requires a two-step request — first obtain a session token (requires a PUT request with a session TTL header), then use that token in subsequent requests. SSRF attacks typically cannot perform the PUT request, blocking the attack. Always enforce IMDSv2 on every EC2 instance.

# Enforce IMDSv2 on a running instance via AWS CLI aws ec2 modify-instance-metadata-options \ --instance-id i-1234567890abcdef0 \ --http-tokens required \ --http-endpoint enabled # Test from inside the instance — IMDSv2 token request TOKEN=$(curl -s -X PUT "http://169.254.169.254/latest/api/token" \ -H "X-aws-ec2-metadata-token-ttl-seconds: 21600") # Use token to get IAM role credentials curl -s http://169.254.169.254/latest/meta-data/iam/security-credentials/MyRole \ -H "X-aws-ec2-metadata-token: $TOKEN" # Returns: AccessKeyId, SecretAccessKey, Token, Expiration (rotate hourly)
💡 IMDSv2 as a Cloud Architect Security Standard

When you become a Cloud Security Architect, one of your first infrastructure standards will be: "All EC2 instances must use IMDSv2." You'll enforce this via an SCP that denies launching instances with http-tokens=optional, and a Config rule that flags any non-compliant instances. This one control would have prevented the Capital One breach at the infrastructure level — before any WAF misconfiguration could be exploited.

EBS — Elastic Block Store Deep Dive
⏱ 1.5 hrs📖 Storage Types + Encryption🔬 EBS Snapshots & Encryption Lab
Lesson

EBS — The Hard Drive for Your EC2 Instance

Elastic Block Store (EBS) provides persistent block storage for EC2 — think of it as the SSD or hard drive attached to your virtual server. EBS persists independently of the EC2 instance — you can stop an instance, start it again, and your EBS data survives. You can also detach an EBS volume and attach it to a different instance, making data recovery and migration straightforward.

EBS is the root volume for your EC2 instance (the boot drive with the OS) and can also be additional data volumes. Most AWS databases (RDS, self-managed MySQL/PostgreSQL on EC2) use EBS for storage.

EBS Volume Types — Performance vs Cost

TypeNameUse CasePerformance
gp3General Purpose SSDBoot volumes, most workloads3,000 IOPS baseline, up to 16,000 IOPS
io2Provisioned IOPS SSDCritical databases (MySQL, Oracle)Up to 64,000 IOPS, sub-ms latency
st1Throughput HDDBig data, log processingHigh throughput, low IOPS, cheap
sc1Cold HDDInfrequently accessed dataLowest cost, lowest performance
EBS Encryption — At-Rest Security

EBS encryption uses AES-256 with keys managed by AWS KMS (Key Management Service). When enabled:

✓ Data at rest on the volume is encrypted ✓ Data in transit between EC2 and EBS is encrypted ✓ Snapshots created from encrypted volumes are encrypted ✓ Volumes restored from encrypted snapshots are encrypted ✓ No performance impact — encryption handled by hardware Enable account-wide default encryption: EC2 Console → Settings → EBS Encryption → Enable by default + select KMS key Policy: EVERY EBS volume must be encrypted in financial services environments
EBS Snapshots — Backup, Recovery & Security Concern

Snapshots are point-in-time backups of EBS volumes stored in S3 (AWS-managed). They're how you back up EC2 instances. The security concern: EBS snapshots can be made public or shared with other AWS accounts.

Attack scenario: Developer creates snapshot of production EBS volume for debugging. Accidentally makes it public. Contains: database files, application configs, API keys. Attack: Attacker discovers public snapshot via AWS snapshot enumeration. Creates new volume from snapshot. Mounts it. Reads all data without any authentication. Prevention: 1. Enable AWS Config rule: "ebs-snapshot-public-restorable-check" 2. Use IAM policy: deny ec2:ModifySnapshotAttribute with createVolumePermission=all 3. Audit: AWS Config → Check for public snapshots monthly
🏢 Financial Services Analogy

EBS encryption for investor data is like PIPEDA-required encryption for financial records stored at rest. If a physical hard drive was stolen from an Equiton server room, encrypted data is unreadable without the KMS key. The KMS key itself is protected by IAM policies and audit logs — providing the full chain of custody required by OSFI B-13 guidelines.

S3 — Object Storage, Access Control & Public Bucket Breaches
⏱ 1.5 hrs📖 S3 Fundamentals + Security🔬 S3 Bucket Configuration Lab
Lesson

S3 — The Most Breached AWS Service

Amazon S3 (Simple Storage Service) stores objects — files, images, backups, logs, datasets — in buckets. S3 is infinitely scalable, 11 nines durable, and the default storage layer for virtually every AWS workload. It's also the source of the majority of high-profile AWS data breaches.

The breach pattern is almost always the same: a developer creates an S3 bucket, sets it to public for convenience, puts sensitive data in it, and forgets. Automated tools scan all S3 buckets continuously — a public bucket with PII is discovered within hours. Equiton would be storing investor financial data in S3 — getting this wrong is a regulatory crisis, not just an IT incident.

S3 Core Concepts

S3 Terminology

Bucket: Top-level container. Globally unique name. Lives in a specific region. e.g., equiton-investor-reports-prod

Object: A file stored in a bucket. Identified by its key (path). e.g., 2024/q4/investor-report-001.pdf

Prefix: Folder-like path within a bucket. S3 is flat storage — "folders" are naming conventions in the key.

Object URL: https://BUCKET.s3.REGION.amazonaws.com/KEY — if the bucket is public, this URL is directly accessible by anyone on the internet.

The S3 Permission Hierarchy — 3 Layers

S3 has three overlapping permission layers that all must allow access for it to succeed:

Layer 1 — IAM Policy: What can this AWS identity do with S3? (identity-based) e.g., Allow s3:GetObject on arn:aws:s3:::equiton-reports/* Layer 2 — Bucket Policy: What identities can access this bucket? (resource-based) e.g., Allow s3:GetObject from VPC endpoint only. Deny all other sources. Layer 3 — S3 Block Public Access: Account/bucket-level override that DENIES all public access regardless of bucket policies. The master switch. → Always ON for private data. Full stop. Access granted only if: IAM allows AND Bucket Policy allows AND Block Public Access is Off If Block Public Access is ON → public access denied regardless of policies
⚠️ S3 Block Public Access — The Control That Prevents Breaches

S3 Block Public Access settings exist at both the account level and the bucket level. Enable it at the account level to prevent any bucket in the account from being made public — this is the single most impactful S3 security control. Any future bucket policy that accidentally grants public access will be blocked by this override.

AWS Console → S3 → Block Public Access settings for this account → Enable all four settings → Save. Do this first for any new AWS account before creating any buckets.

S3 Versioning is enabled on a bucket containing financial reports. A junior developer runs "aws s3 rm s3://equiton-reports/q4-2024.pdf". What happens to the file?
  • A
    The file is permanently deleted immediately
  • B

  • C

For Equiton's financial reports, versioning + lifecycle policies (keep 7 years for PIPEDA) + MFA Delete (require MFA to delete versions permanently) provides complete data protection against both mistakes and malicious deletion.

S3 Encryption — 4 Options, One Right Answer for Financial Data

S3 supports multiple encryption models. As a Cloud Security Architect, you need to know which to use and why — choosing wrong has compliance and audit implications.

OptionKey Managed ByAudit TrailUse for PII?

S3 Bucket Policies — Resource-Based Access Control

Bucket policies are JSON documents attached to a bucket that define who can do what. Unlike IAM policies (which are on the identity), bucket policies are on the resource — they can grant access to cross-account identities, specific VPC endpoints, and can enforce conditions like "only allow if encrypted."

{ "Version": "2012-10-17", "Statement": [ { "Effect": "Deny", "Principal": "*", "Action": "s3:*", "Resource": [ "arn:aws:s3:::equiton-investor-data", "arn:aws:s3:::equiton-investor-data/*" ], "Condition": { "StringNotEquals": { "aws:sourceVpce": "vpce-0123456789abcdef0" } } } ] } → This policy DENIES all S3 access unless the request comes through a specific VPC Endpoint. → Even someone with AdministratorAccess IAM cannot bypass this if not in the VPC. → Data never transits the public internet — stays within AWS network.
S3 Intelligent Tiering & Lifecycle — Cost + Compliance

S3 has multiple storage classes. Lifecycle policies automatically move objects between classes based on age, reducing cost while maintaining required retention:

S3 Standard: Instant access, high cost — active investor reports S3 Standard-IA: Instant access, lower cost — reports from last 90 days S3 Glacier Instant: Instant retrieval, much lower cost — archived reports S3 Glacier Flexible: 3-5hr retrieval, very low cost — compliance archives S3 Glacier Deep Archive: 12-48hr retrieval, lowest cost — 7yr regulatory retention Lifecycle rule for PIPEDA 7-year retention: Day 0: Upload → Standard (active) Day 90: → Standard-IA (infrequent access) Day 365: → Glacier Instant Retrieval (archived, but retrievable quickly if audited) Day 730: → Glacier Deep Archive (7-year regulatory hold) Day 2555: → Expire (delete after 7 years)

Lab 5
Write a Bucket Policy: Enforce Encryption & Add Lifecycle Rules
  • 1
    Open your S3 bucket from Day 4. Go to Properties → Default encryption → Edit. Change from SSE-S3 to SSE-KMS. Key: AWS managed key (aws/s3). Save. All new objects are now encrypted with a KMS key that generates audit trails in CloudTrail.
  • 2
    Add a bucket policy to enforce encryption on upload: Permissions → Bucket policy → Edit. Paste this policy (replace YOUR-BUCKET with your bucket name):
    {"Version":"2012-10-17","Statement":[{"Effect":"Deny","Principal":"*","Action":"s3:PutObject","Resource":"arn:aws:s3:::YOUR-BUCKET/*","Condition":{"StringNotEquals":{"s3:x-amz-server-side-encryption":"aws:kms"}}}]}
    This denies any upload that doesn't specify KMS encryption — prevents unencrypted data from ever entering the bucket.
  • 3
    Test the policy: try uploading a file via the console. It should succeed (console uses KMS by default). The policy prevents programmatic uploads without the encryption header.
  • 4
    Add a lifecycle rule: Management → Lifecycle rules → Create rule. Name: archive-and-retain. Apply to all objects. Add transitions: 90 days → Standard-IA, 365 days → Glacier Instant Retrieval. Add expiration: 2555 days (7 years). Create. This automates PIPEDA retention.
  • 5
    Review S3 Storage Lens (free tier): S3 → Storage Lens → dashboard. See: total storage, object count, encryption coverage %, versioning coverage %. An encryption coverage below 100% means unencrypted data exists — a compliance gap.
  • 6
    Write in notes: What is the difference between SSE-S3, SSE-KMS, and SSE-KMS with CMK? Why does a VPC endpoint bucket policy improve security even if all objects are encrypted?

Q1 of 3
Equiton's OSFI compliance officer asks: "Can you prove exactly who accessed each investor document and when?" Which S3 encryption option enables you to answer yes?
  • A
    SSE-S3 — AWS handles encryption, providing built-in compliance reporting
  • B
    SSE-KMS with a customer-managed key — every decryption operation generates a CloudTrail event with the IAM identity, timestamp, and key used
  • C
    Client-side encryption — you control everything including the audit logs
  • D
    SSE-S3 with versioning enabled — versioning creates the access history
    • ✓ Correct. SSE-KMS with a customer-managed key (CMK) gives you full key usage audit trails in CloudTrail. Every S3 GetObject call triggers a KMS Decrypt API call, which CloudTrail records with: who called it (IAM identity), when (timestamp), from where (IP address), and which key was used. This is the auditor's gold standard — you can prove exactly who read each document and when. SSE-S3 has no KMS audit trail. S3 access logging captures S3-level requests but not the crypto operations.
    Q2 of 3
    A bucket policy statement contains: "Effect": "Deny", "Action": "s3:PutObject", with Condition: s3:x-amz-server-side-encryption must equal "aws:kms". What does this enforce?
    • A
      Only KMS-encrypted IAM users can read objects from the bucket
    • B
      Any upload that doesn't specify KMS encryption is denied — enforces encryption at upload time, preventing unencrypted data from entering the bucket
    • C
      All objects in the bucket will be re-encrypted with KMS automatically
    • D
      The bucket is encrypted at rest — downloads are still unencrypted
    ✓ Correct. This is a "deny upload if not encrypted" policy — a push control that enforces encryption at write time rather than just setting a default. The default encryption setting is a pull control (applies encryption if the uploader doesn't specify) — but a sophisticated misconfiguration could bypass it. The bucket policy deny is stronger: it explicitly rejects any PutObject that doesn't include the encryption header. Together, default encryption + deny-unencrypted policy provides defense in depth for encryption enforcement.
    Q3 of 3
    Why configure an S3 lifecycle policy to move investor documents to Glacier Deep Archive after 1 year, expiring after 7 years, rather than just keeping everything in S3 Standard?
    • A
      Glacier provides better encryption than S3 Standard for long-term storage
    • B
      Glacier Deep Archive is required by PIPEDA for financial documents
    • C
      Dramatically reduces storage costs (up to 95% vs S3 Standard) while maintaining required 7-year retention; automated lifecycle prevents both early deletion (compliance risk) and indefinite retention past legal requirements (privacy risk)
  • D
    Lifecycle policies are only for performance optimization — compliance requirements don't apply to storage classes
    • ✓ Correct. Glacier Deep Archive costs ~$1/TB/month vs S3 Standard at ~$23/TB/month — a 96% cost reduction for archival data. For Equiton holding 7 years of investor documents, this saves hundreds of thousands of dollars at scale. But the compliance angle is equally important: lifecycle rules prevent both failure-to-retain (deleting before 7 years — regulatory violation) and privacy over-retention (keeping data longer than legally permitted — PIPEDA violation). Automated lifecycle management satisfies both sides of the retention equation simultaneously.

    When EBS and S3 Aren't Enough — The Full Storage Menu

    EBS is block storage attached to one EC2 instance. S3 is object storage accessed via API. But some workloads need shared file storage — multiple EC2 instances reading and writing the same files simultaneously. That's where EFS and FSx come in. And when you need to bridge on-premise storage with AWS, Storage Gateway handles that transition.

    ServiceTypeUse CaseKey Security Feature
    EBSBlock (1 instance)EC2 root volume, databasesAES-256 encryption, snapshot audit
    S3Object (API)Backups, static content, data lakeBucket policies, Block Public Access
    EFSShared file (NFS)Web server content, shared configsEncryption at rest + transit, POSIX permissions
    FSx for WindowsManaged Windows File Server (SMB)Windows apps, Active DirectoryAD integration, Windows ACLs
    FSx for LustreHigh-perf parallel (HPC)ML training, genomics, financial quantEncryption, S3 integration
    Storage GatewayHybrid (on-prem + cloud)Backup, migration, hybrid accessEncrypted transfer, deduplication
    S3 GlacierArchive objectLong-term compliance archivesVault Lock (WORM — Write Once Read Many)
    EFS — Elastic File System for Shared Storage

    EFS is NFS (Network File System) as a managed service. Multiple EC2 instances in the same VPC can mount the same EFS file system simultaneously and read/write to it concurrently — like a shared network drive but fully managed, fully elastic, and highly durable.

    Security configuration for EFS: 1. Create EFS in private subnets only (never public-facing NFS) 2. Security group: allow NFS (port 2049) inbound from EC2 SG only — not 0.0.0.0/0 3. Enable encryption at rest (KMS) and in transit (TLS) 4. Use IAM-based access control for access points 5. Access policy: deny mount without TLS (aws:SecureTransport condition) Mount command with TLS (encryption in transit): sudo mount -t efs -o tls fs-0123456789abcdef0:/ /mnt/efs
    S3 Glacier Vault Lock — Regulatory WORM Compliance

    Glacier Vault Lock enables WORM (Write Once Read Many) compliance — once data is written, it cannot be modified or deleted by anyone, including the root account, for a defined period. This satisfies regulatory requirements like OSFI B-13 and PIPEDA that require tamper-proof retention of financial records.

    Use case: Equiton must retain investor transaction records for 7 years, tamper-proof. Configure: S3 Object Lock (Governance or Compliance mode) + 7-year retention period. Result: Even an admin with AdministratorAccess CANNOT delete these objects before 7 years. Only AWS support can override Compliance mode — with very high justification bar. Compliance mode = true WORM (no override possible within organization) Governance mode = admin with special IAM permission can override (audited)
    🏢 Financial Services Analogy

    S3 Glacier with Vault Lock is the digital equivalent of Equiton's legal department locking away physical investor agreements in a tamper-proof vault. Once documents go in under Compliance mode, not even the CEO can order them deleted before the retention period expires. This is exactly the tamper-evident audit trail that OSFI and PIPEDA require for financial transaction records.

    Q1 of 3
    Equiton's legal team requires that all investor transaction records from the past 7 years cannot be modified or deleted by anyone — including IT administrators — to satisfy OSFI audit requirements. Which AWS storage feature addresses this?
    • A
      S3 Versioning — preserves all previous versions of objects
    • B
      S3 Object Lock in Compliance mode with 7-year retention — no entity, including root account, can delete records before retention expires
    • C
      EFS with read-only mount permissions for all users
    • D
      S3 Glacier Deep Archive — archived objects cannot be modified
    ✓ Correct. S3 Object Lock in Compliance mode is the only AWS control that provides true WORM (Write Once Read Many) — immutable storage where no AWS account, no IAM principal, and no root user can delete or modify objects before the retention date. S3 Versioning protects against accidental deletion but can be overridden by admins. Glacier Deep Archive has low cost but doesn't provide the legal immutability guarantee. Compliance mode + Governance mode: Governance allows authorized admins to override (good for accidental lock), Compliance mode has no override within the organization period.
    Q2 of 3
    An application runs on 10 EC2 instances behind a load balancer. All instances need to read and write to the same set of configuration files in real time. Which storage solution is correct?
    • A
      EBS — attach one volume and share it across all instances
    • B
      EFS — shared NFS file system mountable by all EC2 instances simultaneously with consistent read/write access
    • C
      S3 — all instances write to the same bucket
    • D
      Replicate EBS across all instances using a sync script
    ✓ Correct. EFS is shared file storage — multiple EC2 instances can mount the same EFS simultaneously and see each other's writes in real time with POSIX file system semantics. EBS can only be attached to one instance at a time (except Multi-Attach for io2, with limitations). S3 works for objects via API but doesn't behave like a file system — applications would need to be redesigned to use S3 SDK. EFS is the drop-in replacement for a shared network drive in a cloud-native architecture.
    Q3 of 3
    Why should EFS mount targets use a dedicated security group that only allows NFS traffic (port 2049) from EC2 instance security groups, rather than from 0.0.0.0/0?
    • A
      NFS port 2049 is blocked by AWS at the network level — the security group setting doesn't matter
    • B
      Restricting NFS to specific EC2 SGs ensures only authorized application servers can mount the file system — prevents any instance in the VPC (or with VPC access) from mounting and reading all shared files
    • C
      NFS is already encrypted so network restriction is redundant
    • D
      0.0.0.0/0 only applies to internet traffic — VPC-internal traffic is always secure
    ✓ Correct. EFS security groups follow the same principle as RDS and ElastiCache: use SG referencing instead of CIDR ranges. Allow port 2049 (NFS) inbound from the web/app server SG specifically. This means only EC2 instances in that security group can mount the EFS — not a developer's test instance, not a compromised instance in another security group, and not an attacker who somehow accessed the VPC. Encryption in transit (TLS) protects the data channel, but SG restriction controls who can establish the channel at all.

    Build & Document a Hardened EC2 Deployment with Secure S3 Integration

      Verify your EC2 instance is hardened: Check security group — SSH only from your IP.       Attach a least-privilege IAM role: IAM → Roles → Create role. Use case: EC2. Attach only the permissions needed: AmazonS3ReadOnlyAccess (for now). Name: ec2-learning-role-readonly.     • 3
      Test the IAM role from EC2: SSH into instance. Run: aws sts get-caller-identity. Confirm the response shows your IAM role ARN — not a user. Run: aws s3 ls — you should see your bucket list. Run: aws iam list-users — should fail with AccessDenied (your role has no IAM permissions). Least privilege confirmed.
    • 4
      Connect EC2 to your S3 bucket: From your EC2 instance, download a file from your private S3 bucket: aws s3 cp s3://ankit-learning-secure-[name]/test.txt ~/downloaded.txt. This works because your role has S3 read access — and the request stays within AWS network, never traversing the internet.
    • 5
      Enable AWS CloudTrail: CloudTrail Console → Create trail. Name: learning-audit-trail. S3 bucket for logs: create new (name: ankit-cloudtrail-logs-[name]). Enable for all regions. Log file validation: enabled. This records every API call made in your account — your complete audit trail.

    • 7
      Cleanup to avoid charges: Stop (not terminate) your EC2 instance when not using it. A stopped t3.micro doesn't incur compute charges — only EBS storage charges (~$0.10/month for 8GB). You can restart it next time you need it. Keep CloudTrail on — it's free for the first trail.

Monitoring for 48 hours while data is actively exposed is not an acceptable response.
