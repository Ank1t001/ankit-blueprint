TCP/IP — How Data Actually Moves
⏱ 1.5 hrs📖 Lesson🔬 Browser Dev Tools Lab
Lesson

Data Doesn't Just "Go" Somewhere — It's Packaged, Addressed, and Routed

When you open an investor portal, your browser doesn't send one big blob. It breaks your request into small packets, addresses each one, sends them across the internet (sometimes via different routes), and reassembles them at the destination. Every place a packet travels is a potential security interception point — firewalls and security groups make decisions based on these packet properties.

TCP/IP — 4 Layers

Application Layer: Where your apps live — HTTP/HTTPS (web), SMTP (email), SSH (remote access). What humans interact with.
Transport Layer: TCP (reliable, ordered delivery — slower) or UDP (fast, no confirmation — video/gaming).
Network Layer: IP addresses. Routes packets across networks. Where routers and AWS security controls operate.
Link Layer: Physical bits over ethernet/wifi. MAC addresses.

You visit equiton.ca → HTTPS request → TCP breaks into packets → IP routes → bits transmitted → reassembled at server
Ports — What Service is Listening

IP address = the building. Port number = the apartment. Every network service listens on a specific port.

Port 80 = HTTP · Port 443 = HTTPS · Port 22 = SSH · Port 3306 = MySQL · Port 5432 = PostgreSQL · Port 3389 = RDP (Windows remote desktop)
HTTP vs HTTPS

HTTP: plain text — everything visible on the network. Anyone between you and the server can read it.
HTTPS: TLS-encrypted — even if intercepted, content looks like noise. Always use HTTPS for anything sensitive.

HTTP: "Authorization: password123" (readable by anyone)
HTTPS: "%kD9#!mPx..." (encrypted — unreadable without the key)
📦 Physical Mail Analogy

TCP/IP = postal system. Message broken into parcels (packets). Each gets a from-address (source IP), to-address (destination IP), delivery slot (port). Postal network (routers) decides best route — parcels may take different paths and arrive out of order. TCP is registered mail (delivery confirmed). UDP is dropping a flyer in a mailbox — faster, no confirmation of receipt.

💡 Why This Matters

Security Groups in AWS allow or deny traffic based on IP address and port number. You cannot configure network security without understanding ports. Port 22 open to the internet on a production server = attackers attempting brute-force SSH. Port 443 open = normal HTTPS web traffic. This vocabulary underpins every security decision you'll make.

IP Addresses & CIDR Notation
⏱ 1.5 hrs📖 Lesson + Math🔬 Subnet Design Lab
Lesson

Every Device Has an Address — And You Need to Design Address Ranges

IP addresses are how the internet routes traffic to the right destination. When you design AWS VPCs, you define the IP address ranges for the entire network and each subnet. You don't need to be a subnet calculator — but you must understand what CIDR notation means when you see it in AWS, and why private vs public IP ranges matter.

IPv4 Address Structure

An IPv4 address is 4 numbers (0–255) separated by dots. Each number represents 8 bits of a 32-bit address. Total possible: 2³² = ~4.3 billion unique addresses.

192.168.1.100 → written in decimal (human-readable)
In binary: 11000000.10101000.00000001.01100100
Private vs Public IP Ranges

Private IPs are reserved for internal networks — not routable on the public internet. Used in your office, home wifi, and AWS VPCs.
Public IPs are globally unique — routable on the internet.

Three private ranges (memorize for exam and VPC design):
• 10.0.0.0/8 — large internal networks, AWS VPCs
• 172.16.0.0/12 (172.16–31.x.x) — medium networks
• 192.168.0.0/16 — home routers, small offices

Your laptop's IP is probably 192.168.x.x right now — a private address your router assigned. AWS VPCs typically use 10.0.0.0/16.
CIDR Notation — Defining IP Blocks

CIDR (Classless Inter-Domain Routing) defines a range of IP addresses using a base address and a prefix length. The /number tells you how many bits are fixed — the rest define the range.

10.0.0.0/16 → 16 bits fixed (10.0), 16 bits variable → 65,536 addresses (10.0.0.0 to 10.0.255.255)
10.0.1.0/24 → 24 bits fixed (10.0.1), 8 bits variable → 256 addresses (10.0.1.0 to 10.0.1.255)
10.0.1.5/32 → 32 bits fixed → exactly 1 address
CIDRAddressesAWS Use
/1665,536VPC (common choice)
/24256Subnet within VPC
/2816Small subnet (AWS minimum)
/321Specific IP in SG rules
⚠️ AWS Reserves 5 IPs Per Subnet

In every AWS subnet: .0 (network), .1 (VPC router), .2 (DNS), .3 (future use), .255 (broadcast) are reserved. A /24 (256 addresses) gives you only 251 usable IPs. A /28 (16 addresses) gives you only 11. Always account for this when sizing subnets.

🏙️ City Analogy

CIDR is a postal code system. 10.0.0.0/8 = entire province (millions of addresses). 10.0.0.0/16 = city. 10.0.1.0/24 = one street. 10.0.1.5/32 = one house. When designing a VPC, you choose city size (/16) then divide it into neighborhoods (subnets at /24).

DNS — How Domain Names Become IP Addresses
⏱ 1.5 hrs📖 Lesson🔬 nslookup Lab
Lesson

The Internet's Phone Book

Humans remember names. Computers use IP addresses. DNS (Domain Name System) translates one to the other. When you type equiton.ca, DNS resolves it to an IP address — then your browser connects. DNS is also a significant attack surface. DNS hijacking, subdomain takeover, and cache poisoning are real, common attacks that you'll defend against as a Cloud Security Engineer.

DNS Resolution Chain — Step by Step

When you request a domain, this happens in milliseconds:

1. Browser cache — "Do I already know this IP?"
2. OS hosts file — "Any manual override?"
3. Recursive resolver (ISP or 8.8.8.8) — "Do I have this cached?"
4. Root nameserver — "I don't know equiton.ca — ask the .ca server"
5. .ca TLD nameserver — "equiton.ca is handled by nameserver ns1.example.com"
6. Authoritative nameserver — "equiton.ca = 104.21.33.50" ✓
7. Result cached (for TTL seconds) and returned to browser
DNS Record Types — Memorize These

A record: Domain → IPv4 address. equiton.ca → 104.21.33.50
AAAA record: Domain → IPv6 address
CNAME: Alias — one domain points to another. www.equiton.ca → equiton.ca
MX record: Mail Exchange — where to deliver email for this domain
TXT record: Text — used for SPF, DKIM, domain verification
NS record: Nameserver — which server is authoritative for this domain
TTL: Time to Live — how long resolvers should cache this record (seconds)

⚠️ DNS Security Risks You Must Know

DNS Hijacking: Attacker changes your DNS records to redirect traffic to malicious servers. Protect with: registrar MFA, DNSSEC, Route 53 transfer lock.
Subdomain Takeover: A CNAME points to a deleted cloud resource. Attacker claims that resource and controls your subdomain — can serve phishing pages under your domain. Always clean up stale DNS records.
Cache Poisoning: Injecting false records into resolver caches. Mitigated by DNSSEC.

AWS Route 53 — DNS as a Security Control

Route 53 goes beyond basic DNS: Health checks (stop routing to unhealthy endpoints), Failover routing (auto-switch to backup), Geolocation routing (Canadians → ca-central-1), Private hosted zones (DNS resolves only inside your VPC — invisible to public internet).

Private hosted zone: db.internal.equiton → 10.0.3.50 (resolves inside VPC only, unknown to public internet)

VPCs — Your Isolated Private Network in AWS
⏱ 1.5 hrs📖 Lesson🔬 Create Your First Custom VPC
Lesson

A VPC Is Your Private Data Center Inside AWS

A VPC (Virtual Private Cloud) is your own logically isolated section of AWS — a private network you define, control, and segment. Think of AWS's cloud as a massive shared city. Your VPC is your gated private property within that city. You choose the property boundaries (CIDR block), divide it into zones (subnets), control entry/exit (security groups, NACLs), and decide what connects to the outside world (Internet Gateway).

Every resource you create in AWS lives inside a VPC. Network security architecture starts here. AWS creates a Default VPC in every region automatically, but for production financial services workloads, you always build a custom VPC designed for your security requirements.

Internet Gateway (IGW)

The component that allows your VPC to communicate with the internet. Without an IGW, your VPC is completely isolated — nothing can reach it from outside, nothing inside can reach the internet. Attaching an IGW doesn't automatically enable internet access — you still need routes and proper subnet configuration.

VPC with IGW + public subnet route → internet accessible
VPC with IGW + no routes configured → still isolated (IGW does nothing without routing)
NAT Gateway — Outbound-Only Internet for Private Resources

Lets resources in private subnets initiate outbound connections (download patches, call APIs) without being reachable from the internet. Sits in a public subnet. Translates private IP → public IP for outbound requests only. Responses return through the NAT Gateway to the private instance. Inbound from internet is blocked.

Private DB (10.0.3.50) → NAT Gateway → Downloads security patches → ✓
Internet → NAT Gateway → DB (10.0.3.50) → ✗ Blocked
🏰 Castle Analogy

VPC = castle grounds. Outer wall = VPC boundary. Castle gate = Internet Gateway. Drawbridge for servants to leave but no one enters = NAT Gateway. Outer courtyard = public subnet (web servers — guarded but accessible). Inner chambers = private subnets (databases — no public path). Guards at every door = Security Groups.

💡 Default VPC vs Custom VPC

The Default VPC is fine for learning — all subnets are public, which is convenient but insecure for production. For any real workload handling financial data, build a custom VPC: design your own subnet topology, separate public from private, control exactly what has internet access. This is the first architecture decision on every project.

Subnets & Route Tables — Dividing Your VPC
⏱ 1.5 hrs📖 Lesson🔬 Build Subnets in AWS
Lesson

What Makes a Subnet Public or Private? The Route Table.

A subnet divides your VPC into network segments, each in one Availability Zone. The subnet itself is just a CIDR block. What makes it "public" or "private" is purely determined by its route table — the set of routing rules that define where traffic goes.

A subnet with a route sending 0.0.0.0/0 to an Internet Gateway is public. A subnet without that route is private. This isn't a label you apply — it's a consequence of routing configuration. Understanding this distinction precisely is critical for both security architecture and the AWS exam.

Route Table — Routing Rules for a Subnet

Every subnet is associated with a route table. Routes are evaluated in order — most specific match wins.

Public route table:
10.0.0.0/16 → local (all VPC-internal traffic stays inside VPC)
0.0.0.0/0 → igw-abc123 (everything else goes to internet via IGW)

Private route table:
10.0.0.0/16 → local (only intra-VPC traffic — no internet route at all)
Typical 3-Tier Network Architecture

This pattern is used in almost every production AWS environment. Your first portfolio project is based on this.

Public subnet: Load balancer (internet-facing entry point)
↓ traffic passes to:
Private subnet: Application servers (EC2 — no direct internet access)
↓ queries go to:
Private subnet (deeper): Database (RDS — accessible only by app servers)
🏢 Office Building Analogy

VPC = building. Lobby (public subnet) = anyone can walk in from the street. Executive floor (private subnet, app) = need a badge to access. Server room (private subnet, DB) = only IT with special clearance. The elevator routing rules = route tables. Security at each door = Security Groups. Cameras recording traffic = VPC Flow Logs.

💡 Subnet Design for High Availability

Create matching subnet pairs in multiple AZs: public-1a + public-1b, private-app-1a + private-app-1b, private-db-1a + private-db-1b. If one AZ fails, traffic reroutes to the other AZ's matching subnet. This is multi-AZ high availability — standard practice for any production workload at Equiton or anywhere else.

✓ Network-level isolation is the strongest control. There is simply no route from the internet to the private subnet. The attacker's packets have nowhere to go — they can't reach the subnet, let alone the RDS instance. Security Groups (option C) are the second layer — they're evaluated if traffic somehow reaches the resource. But routing is the first and most absolute control. This is why subnet design matters so much.

Security Groups & Network ACLs — Two Layers of Firewall
⏱ 1.5 hrs📖 Critical Lesson🔬 3-Tier Firewall Config
Lesson

Two Network Security Controls — Know Both Cold

AWS provides two firewall-like controls for network traffic. They work at different levels, with different mechanics. You'll configure both on every real project. The AWS exam loves testing the specific differences between them — especially stateful vs stateless and allow-only vs allow/deny.

Security Groups — Instance-Level, Stateful, Allow Only

Level: attached to individual resources (EC2, RDS, Lambda)
Stateful: if inbound traffic is allowed, the return/response traffic is automatically allowed outbound — no explicit rule needed
Rules: allow rules only — you cannot explicitly deny a specific IP
Evaluation: all rules evaluated together — if any rule allows the traffic, it's allowed
Default: all inbound denied, all outbound allowed

SG for web server: Allow inbound 443 from 0.0.0.0/0
SG for database: Allow inbound 5432 only from sg-app (app server Security Group ID — not a CIDR)
Network ACLs — Subnet-Level, Stateless, Allow + Deny

Level: applied to entire subnet — all resources in the subnet are affected
Stateless: you must explicitly allow BOTH inbound AND outbound traffic. Return traffic is not automatic
Rules: allow AND deny rules — you can block specific IPs
Evaluation: rules evaluated in number order, lowest first. First match wins.
Default: default NACL allows all traffic

NACL for public subnet:
Rule 90: DENY all from 203.0.113.5 (known attacker IP)
Rule 100: ALLOW inbound 443 from 0.0.0.0/0
Rule 200: ALLOW outbound 1024-65535 to 0.0.0.0/0 (ephemeral return ports)
FeatureSecurity GroupNACL
LevelResource (instance)Subnet (all resources)
Stateful?Yes — return traffic auto-allowedNo — must allow both directions
Rule typesAllow onlyAllow and Deny
Rule evaluationAll rules, any allow winsLowest rule number first, first match wins
Use caseFine-grained per-resource controlBlock known-bad IPs at subnet level
⚠️ The Stateless NACL Trap — Most Common Mistake

Because NACLs are stateless, allowing inbound HTTPS (443) is NOT enough. You must also allow outbound on ephemeral ports 1024–65535 so the server's TCP response can leave the subnet. Forgetting the outbound ephemeral port rule is the #1 NACL configuration error — traffic appears to arrive but nothing comes back.

🔐 Two-Layer Security Analogy

NACL = building security desk at the entrance. Everyone entering/leaving passes through. Can refuse specific people by name (deny rule). Checks both ways (stateless). Security Group = lock on your individual office door. Even if you got past the lobby, you need permission for this specific room. If you were let in, leaving is automatic (stateful). Both layers work together — neither replaces the other.

Q 1 of 3
A Security Group allows inbound HTTPS (443) from 0.0.0.0/0. A user makes a request. Does the Security Group need an explicit outbound rule for the server's response?
  • A
    No — Security Groups are stateful. Return traffic for an allowed inbound connection is automatically permitted

  • C

✓ Security Groups are stateful. When an inbound connection is allowed, AWS tracks the connection state and automatically permits the response outbound. You don't write matching outbound rules for each inbound rule. This is the opposite of NACLs (stateless) where you must explicitly configure both directions. Stateful vs stateless is the most tested difference between these two controls.
Q 2 of 3
IP 203.0.113.5 is actively attacking Equiton's public subnet. You need to block all traffic from this IP at the network level. Which control do you use?
  • A
    Security Group — add a deny rule for this IP
  • B

✓ NACLs are the right tool because they support explicit DENY rules. Security Groups are allow-only — you cannot add a "deny this IP" rule, only remove allow rules. A NACL rule like: Rule 50 DENY ALL from 203.0.113.5/32 — placed with a low number (evaluated first) — immediately blocks that IP at the subnet boundary before it reaches any resource or Security Group.
Q 3 of 3
You add a NACL inbound rule allowing HTTPS (443). Users still cannot load the website. What is the most likely cause?
  • A
    Security Group is overriding the NACL allow rule
  • B

    HTTPS requires both port 443 and port 80 to be permitted
✓ NACLs are stateless — you must explicitly allow both directions. When a browser connects on port 443, the server responds from an ephemeral port (1024–65535). Always add outbound ephemeral port allowance when configuring NACLs for internet-facing subnets.

Q 1 of 3
An EC2 instance in a public subnet needs to serve HTTPS to the internet. Which THREE components must be correctly configured?
  • A
    Internet Gateway attached + Security Group allowing 443 (Security Group handles the routing)
  • B

  • A

  • B

✓ NACLs have one critical advantage: explicit DENY rules. Security Groups are allow-only — you can only add allows and remove allows. If you need to block a specific malicious IP, you need a NACL. Use both in layers: Security Groups for fine-grained stateful control between resources; NACLs as a stateless subnet-level emergency block for known-bad IPs. They complement each other.
